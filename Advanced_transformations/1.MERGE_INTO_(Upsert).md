# MERGE INTO (Upsert) in Delta Lake
`MERGE INTO` is one of the most important Delta Lake commands.

It is used to:

- UPDATE existing rows
- INSERT new rows
- DELETE rows (optional)

This is called:

> Upsert = Update + Insert

---

# Real-Time Scenario

Imagine you already have a customer table:

| customer_id | name  | city      |
|-------------|-------|-----------|
| 1           | Ravi  | Chennai   |
| 2           | Arjun | Bangalore |

New daily file arrives:

| customer_id | name  | city       |
|-------------|-------|------------|
| 2           | Arjun | Hyderabad  |
| 3           | Kiran | Mumbai     |

What should happen?

- `customer_id = 2` → UPDATE city
- `customer_id = 3` → INSERT new row

This is exactly what `MERGE INTO` does.

---

# Step-by-Step Practice

## STEP 1 — Create Initial Delta Table

```sql
CREATE TABLE customers (
    customer_id INT,
    name STRING,
    city STRING
)
USING DELTA
```

---

## STEP 2 — Insert Existing Data

```sql
INSERT INTO customers VALUES
(1, 'Ravi', 'Chennai'),
(2, 'Arjun', 'Bangalore')
```

---

## STEP 3 — View Existing Table

```sql
SELECT * FROM customers
```

### Output

| customer_id | name  | city      |
|-------------|-------|-----------|
| 1           | Ravi  | Chennai   |
| 2           | Arjun | Bangalore |

---

## STEP 4 — Create Source Updates Table

This table represents new incoming data.

```sql
CREATE TABLE customer_updates (
    customer_id INT,
    name STRING,
    city STRING
)
USING DELTA
```

---

## STEP 5 — Insert Update Data

```sql
INSERT INTO customer_updates VALUES
(2, 'Arjun', 'Hyderabad'),
(3, 'Kiran', 'Mumbai')
```

---

## STEP 6 — Check Update Data

```sql
SELECT * FROM customer_updates
```

### Output

| customer_id | name  | city       |
|-------------|-------|------------|
| 2           | Arjun | Hyderabad  |
| 3           | Kiran | Mumbai     |

---

## STEP 7 — Perform MERGE INTO

```sql
MERGE INTO customers AS target
USING customer_updates AS source
ON target.customer_id = source.customer_id

WHEN MATCHED THEN
UPDATE SET
    target.name = source.name,
    target.city = source.city

WHEN NOT MATCHED THEN
INSERT (
    customer_id,
    name,
    city
)
VALUES (
    source.customer_id,
    source.name,
    source.city
)
```

---

## STEP 8 — Verify Final Output

```sql
SELECT * FROM customers
```

### Final Output

| customer_id | name  | city       |
|-------------|-------|------------|
| 1           | Ravi  | Chennai    |
| 2           | Arjun | Hyderabad  |
| 3           | Kiran | Mumbai     |

---

# What Happened Internally?

| Scenario | Action |
|----------|--------|
| Matching `customer_id` found | UPDATE existing row |
| No matching `customer_id` | INSERT new row |

---

# Key Points About MERGE INTO

- Works only with Delta tables
- Used heavily in:
  - ETL pipelines
  - CDC (Change Data Capture)
  - Incremental loading
  - Real-time streaming pipelines
- Avoids duplicate records
- Supports UPDATE + INSERT + DELETE in one command

---

# Simple Syntax Structure

```sql
MERGE INTO target_table AS target
USING source_table AS source
ON target.id = source.id

WHEN MATCHED THEN
UPDATE SET ...

WHEN NOT MATCHED THEN
INSERT ...
```

---

# Why `USING DELTA` Is Required?

```sql
USING DELTA
```

This tells Databricks to store the table in **Delta Lake format** instead of regular Parquet or CSV format.

Delta format provides:

- ACID transactions
- MERGE INTO support
- Time Travel
- Version history
- Schema enforcement
- Faster updates and deletes

Without Delta format:

- `MERGE INTO` will not work properly
- UPDATE and DELETE operations are limited

# Summary

| Feature | Purpose |
|---------|---------|
| `WHEN MATCHED` | UPDATE existing records |
| `WHEN NOT MATCHED` | INSERT new records |
| `ON` condition | Defines matching key |
| `USING` | Source data |
| `MERGE INTO` | Combines UPDATE + INSERT |
